14、MySQL字符集（乱码问题）

Posted on January 8, 2020 本文总阅读量次

这一小节算是接近MySQL基础部分的尾声了，继续加油。

字符集问题

在创建表时，charset就是标识字符集


1:告诉服务器,我给你发送的数据是什么编码的? character_set_client
2:告诉转换器,转换成什么编码? Character_set_connection
3:查询的结果用什么编码? Character_set_results
如果以上3者都为字符集N, 则可以简写为 set names N

1) mysql Server收到请求时将请求数据从character_set_client转换为character_set_connection
2) 进行内部操作前将请求数据从character_set_connection转换为内部操作字符集,步骤如下
A. 使用每个数据字段的CHARACTER SET设定值；
B. 若上述值不存在，则使用对应数据表的字符集设定值
C. 若上述值不存在，则使用对应数据库的字符集设定值；
D. 若上述值不存在，则使用character_set_server设定值。
3) 最后将操作结果从内部操作字符集转换为character_set_results

查看安装的字符序
mysql> show collation;
查看安装的字符集
mysql> show character set;

乱码形成

1、解码时，规则不对，如用gbk解析utf-8。编码、解码不一致，可修复；

2、字符集在向小容量转换时，导致字节丢失，不可修复。

ASCII—>GB2312—>GBK—>unicode<—>utf-8

附录1

前提
计算机用01组合表示一个字符, 一个字节（能表示255种组合， 2的八次方-1）

只有ASCII码， 每个ASCII码一个字节， 因为计算机发明在美国，人家只考虑了自己的英文字符、数字和一些符号。

然后， 各国开始指定自己的编码，比如中国指定了自己的GB2312, 当然用了两个字节，一个字节是不够的， 还要兼容ASCII。

我国有一套编码，其他国家也弄套编码，大家都弄套编码， 始终得不到同一， 于是， Unicode 应运而生。

所有语言编码统一到一套编码里，就不出现乱码了。

注意， 一般unicode编码都是俩字节的， 偏僻的可能用了四字节。

如果我们需要编码的文本都是英文的， 那么用unicode是不是很浪费，明明一个字节就可以，却用了两字节。

为了节约， 出现了utf-8编码， UTF-8编码把一个Unicode字符根据不同的数字大小编码成1-6个字节，常用的英文字母被编码成1个字节，汉字通常是3个字节，只有很生僻的字符才会被编码成4-6个字节。如果你要传输的文本包含大量英文字符，用UTF-8编码就能节省空间。

在内存里统一使用unicode， 记录到硬盘或者编辑文本的时候都转换成了utf8.

总结

unicode 是统一所有语言的一套编码。

utf-8是基于unicode编码的一种节约字节的编码（相当于时候优化）。
